# Lightweight Secure SDLC

PKM Box Store는 개인 프로젝트 규모의 Pokemon TCG 쇼핑몰이다. 이 문서는 기업용 승인 절차가 아니라, 새 기능을 만들 때 보안 요구사항을 놓치지 않기 위한 최소 운영 방식이다.

## SDLC 단계

| Step | Minimum Action | Output |
| --- | --- | --- |
| 1. 요구사항 정리 | 기능이 다루는 사용자, 데이터, API, 외부 서비스를 적는다. | 기능 메모 또는 이슈 |
| 2. Secure Requirement 확인 | `docs/security-requirements.md`에서 관련 ID를 확인한다. 없으면 새 요구사항을 추가한다. | 관련 SR ID |
| 3. 설계 | 인증, 권한, 소유자 검증, 상태 전이, 금액/재고 검증 위치를 정한다. | 간단한 설계 메모 |
| 4. Threat Modeling | `docs/threat-model.md`에서 같은 흐름의 위협을 확인한다. 새 위협은 추가한다. | 관련 TM ID |
| 5. 구현 | 서버 검증을 우선하고, 프론트 검증은 UX 보조로 둔다. Secret은 환경변수로만 다룬다. | 코드 변경 |
| 6. 테스트 | `docs/security-test-checklist.md`의 관련 항목을 실행하거나 테스트를 추가한다. | 테스트 결과 |
| 7. 배포 | migration, CORS, env var, Toss/S3/SMTP 설정을 확인한다. 실제 secret 값은 기록하지 않는다. | 배포 체크 결과 |
| 8. 운영/변경 기록 | 보안 결정, 남은 갭, 운영 이슈를 문서에 남긴다. | status/gap backlog 업데이트 |

## 새 기능 체크 질문

- 이 기능은 로그인 사용자가 필요한가, 관리자만 필요한가, 공개 API인가?
- 서버에서 현재 사용자와 리소스 소유자를 비교하는가?
- 프론트에서 보낸 금액, 가격, 재고, role, 상태 값을 그대로 믿지 않는가?
- 상태 전이가 허용된 순서로만 일어나는가?
- 실패 응답이 secret, token, stack trace, 과도한 내부 정보를 노출하지 않는가?
- 새 입력값에 `@Valid`, 길이, 형식, enum 검증이 있는가?
- 새 파일 업로드나 URL 입력은 확장자, MIME, 크기, 저장 경로를 제한하는가?
- 새 외부 서비스 키는 `.env`, 코드, 문서, 커밋에 남기지 않는가?
- DB 변경은 Flyway migration으로 관리되고 `JPA_DDL_AUTO=validate`를 통과하는가?
- 보안 테스트 체크리스트와 갭 백로그를 업데이트했는가?

## 예시: 카카오 로그인 추가

- 요구사항: 소셜 로그인은 `SR-AUTH`, `SR-JWT`, `SR-SECRET`, `SR-CORS`에 연결한다.
- 설계: Kakao access token 검증은 서버에서 수행하고, 기존 회원 연결 기준을 정한다.
- 위협: 위조된 provider token, 이메일 불일치, 계정 탈취, redirect URI 오설정을 확인한다.
- 구현: Kakao client secret은 환경변수로만 주입한다.
- 테스트: 잘못된 provider token, 이미 가입된 이메일, 탈퇴/비활성 계정, CORS origin을 확인한다.

## 예시: 상품 이미지 업로드 변경

- 요구사항: `SR-FILE-001`, `SR-ADMIN-001`, `SR-SECRET-001`에 연결한다.
- 설계: 업로드 API는 관리자만 허용하고, 서버에서 확장자/MIME/크기/S3 key를 검증한다.
- 위협: 악성 파일 업로드, 원본 파일명 기반 경로 오염, 공개 버킷 오설정을 확인한다.
- 구현: 원본 파일명은 저장 key에 사용하지 않는다.
- 테스트: 빈 파일, 5MB 초과, MIME 불일치, 비이미지 확장자, 일반 회원 업로드를 확인한다.

## 예시: 결제 흐름 수정

- 요구사항: `SR-PAY-001`, `SR-PAY-002`, `SR-ORDER-001`, `SR-STOCK-001`에 연결한다.
- 설계: 결제 금액과 provider order id는 서버 주문 기준으로 검증한다.
- 위협: 프론트 금액 조작, 다른 사용자 주문 결제, 중복 승인/취소, Toss 응답 조작을 확인한다.
- 구현: 주문 상태 전이와 재고 이력 중복 처리를 명확히 한다.
- 테스트: amount 조작, `paymentKey` 재사용, provider order id 불일치, 중복 승인/취소를 확인한다.

## 운영 원칙

- 문서가 코드보다 앞서지 않아도 된다. 다만 보안 요구사항과 갭은 남긴다.
- 개인 프로젝트에서는 모든 항목을 즉시 자동화하지 않아도 된다. P0 항목부터 테스트로 고정한다.
- 완료된 기능과 확인 필요한 기능을 분리한다.
- 실제 secret 값은 절대 문서에 쓰지 않는다.
